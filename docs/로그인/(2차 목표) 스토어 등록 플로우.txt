
제안하는 로그인/스토어 등록 플로우 (2차 목표)
🔐 정리된 플로우 (고객사 변동 불가 조건 반영)
1. Shane 등록 단계 (AdminDB)
tb_store 에 신규 업체 추가 → StoreID 자동 생성 (예: 1001).
tb_user 에 해당 Store의 최초 AdminUser 생성:
c_email = NULL (아직 없음)
c_password = 임시비번 (랜덤 생성)
i_role = 1 (Admin)
StoreID + 임시PW 조합은 AdminDB에 고정 → 고객사에서 절대 변경 불가.

2. 고객 최초 로그인
입력: StoreID + 임시PW
성공 시 → 추가 등록 절차로 이동:
Email 입력 (고객 고유 ID)
새 Password 설정 (고객 전용 PW)
완료 후 → tb_user에 c_email과 새 PW 저장, c_password는 더 이상 임시PW가 아님.
이후부터는 StoreID + Email + Pw로만 로그인 가능.

3. 고객 일반 로그인
입력: StoreID + Email + Pw
AdminDB 검증:
StoreID 존재 여부 확인 (삭제/중지된 업체인지)
User Email, Pw 확인
JWT 발급 후 Store DB 연결.

4. 보안 효과
StoreID + 임시PW는 고객이 변동 불가 → 최초 인증 키 역할.
고객은 이후 반드시 Email + Pw 등록해야 정상 이용 가능.

무분별하게 Store가 생기거나, StoreID만으로 로그인되는 문제를 원천 차단.


🔑 로그인 & Register 플로우 (최종안)
1. 로그인 시도
사용자가 StoreID + Email + Pw 입력.
AdminDB에서 검증 →

✅ 있으면 로그인 성공
❌ 계정이 없으면 → "Register" 버튼 노출

2. Register 절차
고객이 "Register" 클릭 → 입력폼:
StoreID, StorePW (Shane이 업체 등록 시 발급한 고정 비밀번호)

👉 여기서 StoreID + StorePW 조합이 맞는지 AdminDB에서 검증.
✅ 검증 성공 시 → 새 계정 생성 가능
Email, Password, 이름, 전화번호 등 기본정보
Role (기본: 일반 사용자, 또는 AdminUser가 추가 승인 필요)

❌ 검증 실패 시 → "등록 불가" 에러

3. 등록 이후 로그인
이제 고객은 StoreID + Email + Pw 로 로그인.
StorePW 는 오직 신규 Register 용도이고, 로그인 시에는 사용하지 않음.
즉, StorePW = 업체 고유 인증키 같은 역할.

🚀 핵심 포인트
StorePW = 고객사가 직접 바꾸지 못하는 마스터 키 (Shane이 발급).
신규 계정 생성 시 반드시 StorePW 필요 → 무분별 계정 생성을 원천 차단.
이후 로그인은 Email 기반 → 일반 SaaS 계정과 동일하게 동작.

🔑 중복 로그인 처리 (자동 로그아웃 포함)
1. 새로운 로그인 시도
사용자가 StoreID + Email + Pw 입력
서버에서 해당 User의 active 세션이 있는지 확인
만약 있으면 → 메시지 반환:
이미 다른 곳에서 사용 중입니다.
기존 세션을 종료하고 로그인하시겠습니까?
클라이언트가 "승인" 선택 시 → force=true로 다시 요청

2. 승인 후 동작
서버는 기존 세션을 terminated로 변경
동시에 기존 클라이언트에 WebSocket/Push 알림을 보냄:
다른 기기에서 로그인하여 자동 로그아웃 되었습니다.
기존 클라이언트는 세션 만료 처리 → 로그인 화면으로 이동
새로운 로그인 요청자에게 새 JWT 토큰 발급

🔑 로그인 관련 테이블 설계 (확정안)
-- 1️⃣ 업체 관리 테이블
CREATE TABLE IF NOT EXISTS tb_store (
    i_store_id     INTEGER PRIMARY KEY AUTOINCREMENT,  -- Store ID (1001, 1002 …)
    c_store_name   TEXT NOT NULL,                      -- 업체명
    c_store_pw     TEXT NOT NULL,                      -- Store 고유 비밀번호 (Shane 발급, 변경 불가)
    c_status       TEXT DEFAULT 'active',              -- 상태 (active, inactive)
    dt_created     DATETIME DEFAULT CURRENT_TIMESTAMP  -- 생성일
);

-- 2️⃣ 사용자 계정 테이블
CREATE TABLE IF NOT EXISTS tb_user (
    i_user_id    INTEGER PRIMARY KEY AUTOINCREMENT,    -- 사용자 ID
    i_store_id   INTEGER NOT NULL,                     -- 소속 Store
    c_email      TEXT NOT NULL,                        -- 사용자 Email (로그인 ID)
    c_password   TEXT NOT NULL,                        -- 해시된 비밀번호
    i_role       INTEGER DEFAULT 2,                    -- 권한 코드 (1=Admin, 2=User)
    c_status     TEXT DEFAULT 'active',                -- 상태 (active, inactive, locked)
    dt_created   DATETIME DEFAULT CURRENT_TIMESTAMP,   -- 생성일
    FOREIGN KEY (i_store_id) REFERENCES tb_store(i_store_id),
    UNIQUE(i_store_id, c_email)                        -- 동일 Store 내 Email 중복 방지
);

-- 3️⃣ 로그인 세션 테이블
CREATE TABLE IF NOT EXISTS tb_session (
    i_session_id     INTEGER PRIMARY KEY AUTOINCREMENT, -- 세션 ID
    i_user_id        INTEGER NOT NULL,                  -- 사용자 ID
    c_jwt_token      TEXT NOT NULL,                     -- JWT 토큰
    dt_login         DATETIME DEFAULT CURRENT_TIMESTAMP,-- 로그인 시간
    dt_last_active   DATETIME DEFAULT CURRENT_TIMESTAMP,-- 마지막 활동 시간
    c_status         TEXT DEFAULT 'active',             -- active / terminated
    dt_terminated    DATETIME,                          -- 종료 시각
    FOREIGN KEY (i_user_id) REFERENCES tb_user(i_user_id)
);
